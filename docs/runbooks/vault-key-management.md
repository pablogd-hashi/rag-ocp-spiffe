# Vault Key Lifecycle Management — Operational Runbook

## Overview

This runbook covers centralised cryptographic key lifecycle management using
Vault Enterprise's Key Management secrets engine (for cloud KMS distribution)
and KMIP secrets engine (for on-premises key management protocol integration).

Both require the Advanced Data Protection (ADP) license module.

Reference: HashiCorp Validated Design — Vault Operating Guide for Scaling.

---

## Key Management Secrets Engine

### Supported Cloud Providers

| Provider | Integration |
|----------|-------------|
| AWS KMS | Native |
| Azure Key Vault | Native |
| GCP Cloud KMS | Native |

### Enable the Engine

```bash
vault secrets enable keymgmt
```

### Key Lifecycle

#### 1. Create a Key

```bash
vault write -f keymgmt/key/example-key type="rsa-2048"
```

#### 2. Configure a KMS Provider

Example for Azure Key Vault:

```bash
vault write keymgmt/kms/azure-prod \
  provider="azurekeyvault" \
  key_collection="my-keyvault" \
  credentials=client_id="${CLIENT_ID}" \
  credentials=client_secret="${CLIENT_SECRET}" \
  credentials=tenant_id="${TENANT_ID}"
```

#### 3. Distribute Key to KMS

```bash
vault write keymgmt/kms/azure-prod/key/example-key \
  purpose="encrypt,decrypt" \
  protection="hsm"
```

- **`purpose`** — cryptographic capabilities: encrypt, decrypt, sign, verify,
  wrap, unwrap.
- **`protection`** — `hsm` for hardware-backed operations, `software` for
  software-backed.

#### 4. Rotate

```bash
vault write -f keymgmt/key/example-key/rotate
```

Creates a new key version in both Vault and the distributed KMS provider. The
new version becomes the current version for all cryptographic operations.

#### 5. Disable Old Versions

```bash
vault write keymgmt/key/example-key min_enabled_version=2
```

All versions below `min_enabled_version` are disabled. Setting this to `0`
re-enables all versions.

#### 6. Remove from KMS

```bash
vault delete keymgmt/kms/azure-prod/key/example-key
```

Deletes the key from the cloud KMS provider. The key persists in Vault's
secrets engine and can be redistributed later.

### BYOK Benefits

- Generate keys with trusted entropy sources independent of the cloud provider.
- Retain exclusive organisational ownership and control of key material.
- Maintain offsite copies for disaster recovery.
- Seamless portability across on-premises and multi-cloud environments.

### Timeout Handling

If a timeout occurs when distributing a key to a KMS provider, increase the
value of `VAULT_CLIENT_TIMEOUT` before retrying.

## KMIP Secrets Engine

### Protocol

OASIS Key Management Interoperability Protocol (KMIP) v1.4. The KMIP server
component listens on a dedicated TCP port (default 5696) and authenticates
clients via TLS certificates generated by Vault's internal KMIP CA.

### Supported Profiles

- Baseline Server
- Symmetric Key Lifecycle Server
- Basic Cryptographic Server
- Asymmetric Key Lifecycle Server
- Advanced Cryptographic Server

### Scopes and Roles

- **Scopes** partition KMIP object storage into named buckets.
- **Roles** define the allowed operations per scope.
- Access control is enforced via role-based TLS certificates — Vault ACL
  policies do not apply to KMIP operations.

### Supported Operations

activate, create, create_keypair, decrypt, destroy, encrypt, get, import,
locate, mac, mac_verify, register, rekey, rekey_keypair, revoke, sign,
signature_verify, and more. Use `operation_all` to grant all capabilities or
`operation_none` to deny all.

### Platform Integrations

| Platform | Use case |
|----------|----------|
| MongoDB | Field-level encryption |
| IBM Storage Scale | RKM key server |
| AWS KMS External Key Store (XKS) | PKCS#11 provider integration |

### Debugging

Enable detailed KMIP logging before Vault startup:

```bash
export KMIP_CONNECTION_DEBUGGING=1
export KMIP_REQUEST_DEBUGGING=1
export KMIP_RESPONSE_DEBUGGING=1
```

### Load Balancer Notes

- KMIP listeners are enabled only on performance standby servers.
- Without performance standby, route all KMIP traffic to the cluster leader.
- Use pass-through mode — TLS must not be terminated by the load balancer.
- Support long-lived connections; round-robin routing is acceptable.

## Best Practices

1. **Rotation schedule** — establish rotation cadence based on risk assessment
   and regulatory requirements. Maintain at least two active key versions
   during transitions.
2. **Monitoring** — track key usage patterns via audit logs. Set up alerts for
   anomalous access patterns.
3. **Compliance** — Vault audits all KMS and KMIP operations. Analyse audit
   trails regularly for regulatory compliance (PCI-DSS, GDPR, HIPAA).
4. **Revocation** — promptly deactivate and replace compromised keys.
